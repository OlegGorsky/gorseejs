# Ambiguity Policy

This document defines how Gorsee removes ambiguous guidance from docs and scaffolds as a mature product.

## Rules

- prefer one clear path over multiple half-supported options
- avoid wording that implies support where validation does not exist
- avoid scaffolds that suggest optional complexity before teams need it
- when two docs overlap, one must become canonical and the other should point to it

## Ambiguity Signals

The following are signs of drift:

- docs describing the same task in materially different ways
- examples demonstrating competing defaults
- scaffolds that imply compatibility-only or non-canonical paths
- product language that claims “flexibility” without a clear recommended path

## Product Rule

Ambiguity is not neutral. For an AI-first framework, ambiguity is a product defect.
