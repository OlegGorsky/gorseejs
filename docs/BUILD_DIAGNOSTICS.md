# Build Diagnostics

This document defines build and asset diagnostic expectations for Gorsee as a mature product.

## Purpose

Build diagnostics are not optional developer convenience. They are part of the runtime contract for a mature product.

Teams and agents must be able to answer:

- which route bundle is missing or drifted
- whether the client manifest is incomplete
- whether a prerendered route was emitted
- whether deploy artifacts still contain placeholder origins
- whether release smoke exercised the same public surfaces that production uses

## Required Diagnostic Signals

At minimum, build and release tooling must make it obvious when any of the following fail:

- missing client bundle for a route
- hashed route bundle resolution drift
- missing or stale prerender metadata
- deploy artifact placeholder values such as `REPLACE_WITH_APP_ORIGIN`
- release smoke regressions in scaffolded apps or canonical examples

## Artifact Expectations

The framework must continue to treat these artifacts as first-class build outputs:

- `dist/manifest.json`
- route-level hashed client bundles under `dist/client/`
- prerender metadata emitted during `gorsee build`
- deploy artifacts generated by `gorsee deploy`

`dist/manifest.json` is a versioned machine contract. It must include `schemaVersion`, and runtime loading is required to fail closed when the schema does not match the expected version.

If any of those artifacts drift, diagnostics should identify the concrete artifact, not only emit a generic failure.

## Structured Failure Contract

Build backend failures must surface structured diagnostics instead of opaque backend noise.

At minimum, a client-build diagnostic line should identify:

- `backend`
- `phase`
- `severity`
- `code`
- `file` when available
- `plugin` when available
- the human-readable failure message

Canonical examples:

- `[backend=rolldown phase=bundle severity=error code=ROLLDOWN_BUILD_FAILURE file=routes/index.tsx] Unexpected token`
- `[backend=bun phase=bundle severity=error code=BUN_BUILD_FAILURE] Could not resolve gorsee/runtime`

`gorsee build` is required to print structured backend diagnostics before failing, and AI diagnostics should carry the same failure summary through the build surface.

## Operator Expectations

When a build/runtime incident happens, operators should be able to inspect:

1. `gorsee check`
2. `gorsee build`
3. `dist/manifest.json`
4. generated deploy files
5. `.gorsee/ai-events.jsonl`
6. `.gorsee/agent/latest.json`

## Non-Goals

- Build diagnostics are not a replacement for benchmark evidence.
- Source maps are useful, but they do not replace explicit manifest and artifact validation.
- Release smoke is not allowed to silently skip failure-prone public surfaces.
